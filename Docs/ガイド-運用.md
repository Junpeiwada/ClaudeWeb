# 運用ガイド

## HTTPS化（Tailscale Serve）

iPhoneからAgentNestにHTTPSでアクセスするための設定。
Screen Wake Lock API（AI応答中のスリープ防止）にはセキュアコンテキスト（HTTPS）が必要。

### 背景

- iPhoneからTailscale経由で `http://100.x.x.x:3000` でアクセスしている
- HTTPではScreen Wake Lock APIが使えない（セキュアコンテキスト必須）
- Tailscale Serveを使えばAgentNestのコード変更なしでHTTPS化できる

### 仕組み

```
iPhone Safari
  ↓ HTTPS (443)
Tailscale Serve（TLS終端・リバースプロキシ）
  ↓ HTTP (localhost:3000)
AgentNest Express サーバー
```

- Tailscale が TLS 証明書を自動管理（Let's Encrypt）
- Express サーバーは HTTP のまま変更不要
- アクセスは Tailnet 内のデバイスのみ（インターネットには公開されない）

### 前提条件

- macOS に Tailscale アプリがインストール済み
- Tailscale にログイン済み
- AgentNest サーバーが `localhost:3000` で起動していること（Serve自体は独立して動くが、アクセス時に必要）

### コマンド

CLIのパス: `/Applications/Tailscale.app/Contents/MacOS/Tailscale`

#### 有効化

```bash
/Applications/Tailscale.app/Contents/MacOS/Tailscale serve --bg 3000
```

`--bg` をつけるとバックグラウンドで永続化される（Mac再起動後も有効）。

#### 状態確認

```bash
/Applications/Tailscale.app/Contents/MacOS/Tailscale serve status
```

#### 無効化

```bash
/Applications/Tailscale.app/Contents/MacOS/Tailscale serve off
```

### iPhoneからのアクセス

有効化後、iPhoneのSafariから以下のURLでアクセス:

```
https://junpeimac-mini.tail34f489.ts.net
```

- ポート指定不要（HTTPS標準の443）
- `http://100.113.106.109:3000` は引き続き使えるが、HTTPなのでWake Lock不可

### Tailnet での Serve 有効化

初回のみ、Tailscale の管理画面で Serve 機能を有効にする必要がある:

```
https://login.tailscale.com/admin/machines → 対象マシン → Serve を有効化
```

### ライフサイクル

| 状態 | Tailscale Serve | AgentNest |
|---|---|---|
| 初回セットアップ | `serve --bg 3000` を実行 | 通常通り起動 |
| 通常運用 | 常駐（自動起動） | 必要時に起動 |
| AgentNest停止中 | 動き続ける（502エラーを返す） | 停止中 |
| Mac再起動後 | 自動で復帰 | 手動またはTauriアプリで起動 |
| 不要になった時 | `serve off` で完全停止 | 影響なし |

### 依存関係

```
Wake Lock API（スリープ防止）
  └─ セキュアコンテキスト（HTTPS）が必要
       └─ Tailscale Serve がHTTPS化を担当
            └─ Tailscale アプリ（macOS）
            └─ AgentNest Express サーバー（HTTP, localhost:3000）
```

- Tailscale Serve と AgentNest は独立して動作する
- Tailscale Serve を止めても AgentNest には影響なし（HTTPアクセスは引き続き可能）
- AgentNest を止めても Tailscale Serve には影響なし（502が返るだけ）

## ユニバーサルバイナリ対応

### 現状

現在のリリースは**ビルドしたMacのアーキテクチャ専用**のバイナリを生成する。

- Arm Mac（M1/M2/M3/M4）でビルド → `aarch64-apple-darwin` 専用
- Intel Macでビルド → `x86_64-apple-darwin` 専用

### 影響を受ける箇所

| 箇所 | 内容 |
|------|------|
| Tauriバイナリ（Rust） | `npx tauri build` がホストアーキテクチャ向けにコンパイルする |
| sharpネイティブモジュール | `build-server.mjs` でアーキテクチャ別パッケージをコピー（`@img/sharp-darwin-arm64` / `@img/sharp-darwin-x64`）。存在するもののみコピーされる |
| latest.json | `release.sh` が `darwin-aarch64` と `darwin-x86_64` の両方に同じバイナリURLを記載している。実際は片方のアーキテクチャしか動作しない |

### ユニバーサルバイナリ対応の手順（未実施）

両アーキテクチャ対応のバイナリを作る場合、以下の対応が必要。

#### 1. Rustクロスコンパイルターゲットの追加

Arm Macで作業する場合：

```bash
rustup target add x86_64-apple-darwin
```

Intel Macで作業する場合：

```bash
rustup target add aarch64-apple-darwin
```

#### 2. Tauriビルドコマンドの変更

```bash
npx tauri build --target universal-apple-darwin
```

これにより `aarch64` と `x86_64` の両方をコンパイルし、macOSのユニバーサルバイナリ（fat binary）として結合される。

#### 3. sharpの両アーキテクチャバンドル

`build-server.mjs` の `packagesToCopy` には既に両方記載済み：

```js
const packagesToCopy = [
  "@anthropic-ai/claude-agent-sdk",
  "@img/sharp-darwin-arm64",
  "@img/sharp-darwin-x64",
];
```

ただし `npm install` は通常ホストアーキテクチャのパッケージのみインストールする。両方をバンドルするには：

```bash
npm install --cpu=arm64 @img/sharp-darwin-arm64
npm install --cpu=x64 @img/sharp-darwin-x64
```

#### 4. release.shの変更

ユニバーサルバイナリの場合、成果物のパスが変わる：

- 現在: `src-tauri/target/release/bundle/`
- 変更後: `src-tauri/target/universal-apple-darwin/release/bundle/`

`release.sh` 内の `BUNDLE_DIR` を更新する必要がある。

#### 5. latest.jsonの整合性

ユニバーサルバイナリであれば、`darwin-aarch64` と `darwin-x86_64` に同じURLを指定する現在の構成で正しく動作する。

### 備考

- Rosetta 2が有効なArm Macであれば、x86_64バイナリも動作する（パフォーマンスは劣る）
- 逆にIntel Macではarm64バイナリは動作しない
- 現在はArm Macでリリースしているため、Intel Macユーザーは実質サポート外
